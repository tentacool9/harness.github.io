LLM Harness Design Patterns

Cracked Harness Patterns: Extract Maximum Brain Power from Your LLM

Several open-source LLM harnesses, for ARC-AGI, competition math, web QA, code contests, Minecraft, and multi-hop QA, converge on a small set of design patterns that lift a model like GPT-5 way past what the model alone can do. This page names the patterns and shows how each harness instantiates them.

Each pattern comes with what it helps with, when to use it, when not to, an archetype diagram of the abstract move, and three or four concrete instances drawn from harnesses that build differently around the same idea.

Why patterns, not harnesses. Most writeups of these systems walk through one harness at a time, which buries the design moves under harness-specific details. Reading them side by side, the same moves keep recurring in different dialects. The page below names those moves at the level of abstraction at which they actually port between domains, "multiple experts aggregated by a verifier" rather than "MCTS multi-seed trees" or "code-execution voting." Each pattern is admitted only when several harnesses instantiate it in materially different ways. A harness cheatsheet at the bottom links each system to every pattern it appears in.
Pattern 1

Ensemble of Solvers

Produce many candidates with engineered variation, in parallel or sequentially across iterations; score each with an oracle or executor; aggregate by votes, scores, or a strict ratchet. Never aggregate by the LLM's own confidence.

What it helps with

Turning a frozen API model into an ensemble without retraining. The LLM's self-reported confidence is a poor signal, and N copies of the same call are correlated. An external verifier turns "many guesses", whether produced in parallel or sequentially, into a usable distribution to aggregate over.

When to use it

Tasks where (a) you can produce many candidates (in one shot or across iterations) and (b) there is a deterministic verifier, code execution, exact-match grading, a simulator, oracle outputs, or a near-verifier the harness can build (a fabricated test suite, an LLM-as-critic with process-graded few-shots).

When NOT to use it

Open-ended generation with no oracle (poetry, free-form summarization without a fixed reference). Latency-bound single-call use cases. Tasks where verification is as expensive as solving, so the cost saving of cheap-verifier ensembling disappears.

Concrete instances

Poetiq: engineered diversity, then output-space majority vote

ARC-AGI grid-transformation puzzles· public SOTA, 2 attempts/task· Gemini 3 / GPT-5.1 / Claude / Grok (frozen API)
How this harness uses it

Run a wide flat ensemble of independent expert calls, each producing a Python transform(grid). Diversify them up front with three composable knobs: shuffle_examples, worst-to-best improving_order, and Bernoulli sampling of past attempts shown in the prompt. Execute every candidate in a sandbox; bucket by the canonical hash of the test output; vote by bucket size; pick the top two distinct outputs for ARC's two-attempt slot.

Problem it solved here

ARC awards correctness on the test grid; how the code was derived is irrelevant. Voting in artifact space (edit distance, AST similarity) splits one consensus across many semantically- equivalent code shapes. Naive parallel calls also collapse to correlated samples, so the diversification knobs are mandatory for the vote to mean anything.

 ARC's two allowed attempts"]:::accent

    n --> sb
    sb --> bucket
    bucket --> top

AlphaCodium: iterate-and-keep-best by pass count (strict ratchet)

CodeContests competitive programming· beat prior published results on CodeContests· GPT-4 (frozen, no fine-tuning)
How this harness uses it

Produce many code candidates over a multi-stage pipeline. Score each candidate by how many tests it passes, public tests cheaply, LLM-fabricated tests as a second tier. Keep an explicit passed_tests set. A candidate or edit is accepted only if it strictly grows the set; otherwise revert and try again. The "best so far" is the one whose pass-set is the strict superset of all others, a monotone aggregation.

Problem it solved here

Iterative fix loops drift: patch test A, break test B, patch B, break A. Pass count wobbles; "improving" is illusory. Aggregating by the executor's verdict, with a strict ratchet, makes forward progress measurable. The verifier (test suite) is doing the aggregation, not the LLM's opinion of whether the new code looks better.

Pattern 2

Separation of Creativity and Critique

Re-role a single API model into solver / critic / value function / test author / curriculum by varying only the system prompt and the few-shot block. No second model, no fine-tuning.

What it helps with

Getting "two heads" without two models. A solver and an independent critic give complementary signal even when they share weights, the system prompt sets the cognitive frame, and process-graded few-shots can train the critic to grade process not outcome.

When to use it

Frozen-API settings where you can't fine-tune a separate verifier. Cases where role independence (an honest second opinion, a process scorer, a test author, a curriculum proposer) matters more than role specialization. Especially powerful when one role decomposes into independent checks that scale monotonically with budget.

When NOT to use it

When the verifier needs domain knowledge the solver doesn't have, then you need a separately-trained model, not a re-prompt. When the model's mistakes are so model-architectural (a knowledge cutoff, a tokenization quirk) that re-prompting can't escape them. When you have a real deterministic verifier and don't need LLM-as-critic at all.

LATS: value-function prompt parsed back into a scalar

HotPotQA · HumanEval/MBPP · WebShop· MCTS over LLM trajectories· GPT-4 (frozen API)
How this harness uses it

Wrap the same frozen model behind a value prompt: "Given this trajectory, rate correctness 1-10. Thus the score is __." Parse the completion by substring scanning "10" first, then "9", down to "1"; map to [0.1, 1.0]. The crucial detail: the few-shot block mixes incomplete-but-on-track trajectories scored 10 with wrong-direction trajectories scored 3-4, training the same model to grade process rather than whether the final answer is yet present. Cached per problem so revisited nodes don't re-query.

Problem it solved here

MCTS needs node values, but training a value head requires data, trajectories, and a model you own, none of which exist on top of a frozen API. Without value scoring, tree search collapses into random rollouts. The prompted value function trades a small token cost for a portable scorer that can be reframed (process vs. outcome) by editing the few-shot.

    prompt["Wrap it in a value prompt with hand-picked examples: 
 'incomplete-but-on-track trajectories scored 10, 
 wrong-direction trajectories scored 3-4'"]:::accent
    same["The same frozen language model 
 (no separate value head, no fine-tuning)"]:::muted
    out["Model writes: '…the correctness score is 8'"]
    parse["Scan the output for the first digit: 
 10 first, then 9, down to 1. 
 Map that to a number between 0.1 and 1.0."]:::accent
    why["The few-shot teaches the same model to grade

 rather than the outcome (does it contain the 
 final answer yet?)"]:::note

    traj --> prompt
    prompt --> same
    same --> out
    out --> parse
    why ~~~ prompt

deepsearch-tts: solver and verifier, same loop, different system prompts

BrowseComp / GAIA / xBench web-grounded QA· test-time-scaling gains across multiple open-weights models· Qwen / DeepSeek-R1 / Kimi
How this harness uses it

One generic agent-loop implementation, parameterized on system_prompt, messages, and a tool set. Instantiate it twice: once with a solver prompt (exploratory, breadth-first, "find the answer"), once with a verifier prompt (narrow, condition-by-condition, "check each claim against evidence"). Same tools, optionally the same model. The asymmetry is entirely in the prompt, and empirically the verifier reaps almost all the test-time-scaling benefit, because its work decomposes into independent per-condition checks.

Problem it solved here

Web QA can't trust the solver alone, it will confidently quote stale pages and hallucinate dates. But more budget makes the solver worse past a certain point: over-exploration, contradiction, drift. A separately-trained verifier model would double the surface area for bugs. Prompt-only re-roling gives the asymmetric solver/verifier split with no new code path.

    tools --> budget

AlphaCodium: LLM as test-case author against its own code

CodeContests competitive programming· beat prior published results on CodeContests· GPT-4 (frozen)
How this harness uses it

A dedicated "test-author" prompt asks the model to fabricate 6 additional input/output pairs, sorted easy-to-hard, explicitly including a "large input" case, that go beyond the 1-3 public tests shipped with the problem. The fabricated tests become a second verifier tier behind the public ones: cheap crash-gate first, generated tests second. The same model generates the code and writes the tests it has to pass, in separate calls with separate prompts.

Problem it solved here

CodeContests has weak baseline test suites and lots of distractor solutions that pass the 1-3 visible cases but fail hidden grading. The model is much better at generating hard input cases than at solving the whole problem end-to-end; this prompt-role split exploits that asymmetry, generation budget moves to the verifier.

Pattern 3

Natural Language Reinforcement Learning

The "gradient" between attempts is a natural-language reflection on what went wrong, fed back into the next attempt's prompt. No weight update, the same frozen model is its own in-context tutor.

What it helps with

Improving across attempts on a frozen model. Captures qualitative signal ("I forgot the empty-list case", "the page I read was out of date") that a scalar reward can't express. The model's own retrospective prose is in a format it natively parses.

When to use it

When failure is informative, the model can describe what went wrong in NL. When retries are cheap (you can afford many attempts per task). When the task has a clear pass/fail signal so the reflector has something concrete to react to.

When NOT to use it

When failures are noisy or non-diagnostic (the model can't say what went wrong because it doesn't know, e.g. tasks where its error is silent). Single-shot tasks with no second attempt. Don't run unbounded, stale reflections describe an agent that no longer exists, and accuracy regresses; bound the reflection memory.

    paragraph["The model writes a short natural-language paragraph

 'I retrieved the right page but used outdated data; 
 next time check the most recent entry.'"]:::accent

Concrete instances

Reflexion: Actor / Evaluator / Reflector with NL reflection as cross-attempt memory

HumanEval / LeetCode · AlfWorld · HotPotQA · WebShop· foundational paper for self-reflective agent loops· GPT-4 / GPT-3.5 (frozen)
How this harness uses it

Three role-specialized prompts: Actor produces solutions; Evaluator decides pass/fail deterministically where possible (Python sandbox, exact-match, env reward); Self-Reflector, on failure, writes a short paragraph diagnosing what went wrong and proposing a different approach. The reflection is then injected verbatim into the next Actor prompt. Reflexion is the canonical instantiation of this pattern, almost every later harness with a reflection mechanism descends from it.

Problem it solved here

Naive retry-on-failure is amnesiac: attempt k+1 has no signal that attempt k happened, and the model re-discovers the same dead-ends. Scalar reward back-propagation works for trained agents but not for a frozen black-box LLM. NL reflection is the simplest workable substitute, it lets the same model "train itself" across trials with no weight update at all.

LATS: extends verbal RL to a SECOND head (the value function)

HotPotQA · HumanEval/MBPP · WebShop· MCTS over LLM trajectories· GPT-4 (frozen)
How this harness uses it

Reflexion's contribution was "NL reflection updates the actor." LATS's verbal-RL contribution is that the same NL reflection ALSO updates the value head: it gets spliced into the value-scoring prompt with the directive "this trajectory is incorrect as {reflection}, thus correctness score is 1." So one NL update propagates to two heads at once, sampling diverts away from the dead-end pattern, AND scoring grades similar partial paths low. The gradient analog targets two policies, not one.

Problem it solved here

In MCTS, a memoryless value function keeps scoring confidently-wrong subtrees highly forever, fixing only the actor isn't enough, because the value function will steer rollouts right back into the bad region. Updating both heads with the same NL signal closes the loop: sampling AND scoring move together when the harness learns something. This is the structural reason LATS works on long HotPotQA and WebShop trajectories where actor-only verbal RL stalls.

Voyager: Critic Agent's NL diagnosis as the policy-update payload

open-ended Minecraft, no terminal state· outperformed prior agents on novel-item discovery· GPT-4 / GPT-3.5
How this harness uses it

When the Action Agent's JS code fails (env returns an error, the Critic Agent says success=False), the Critic produces a natural-language diagnosis, e.g. "you tried to craft an iron pickaxe but you don't have a furnace; smelt iron ore first." That diagnosis is injected, verbatim, into the Action Agent's next code-generation prompt. The NL paragraph is the gradient: it's what makes attempt k+1 different from attempt k. Raw env errors come along as additional signal, but the Critic's reasoned prose is what's structurally Reflexion-shaped, a reflective LLM producing the update payload that drives the next attempt.

Problem it solved here

Minecraft is open-ended, no oracle, no terminal state, no "the answer is 42." The only signal the agent gets is what the environment says after its code ran. Verbal RL on top of that signal lets the same frozen model improve its code across attempts without any training step: the Critic's NL diagnosis is what turns "compilation failed" into the actionable next-attempt guidance the Action Agent's prompt can absorb.

Pattern 4

Tree Search: Far and Wide

Search a tree of partial trajectories. At each step, score the partial state with a critic or verifier, then spend more compute on the promising branches and kill the dead ones. Two flavors of this move show up across harnesses: depth-first with explicit tree memory (Monte Carlo Tree Search), and breadth-first with population resampling (Sequential Monte Carlo).

What it helps with

Tasks where the right answer is buried in a large but structured search space, and a flat ensemble wastes compute on rollouts that already wandered off in the first step. Tree search reallocates compute mid-flight: dead branches are killed before they reach a terminal answer the verifier would have rejected anyway, and the surviving branches inherit the freed budget.

When to use it

When you can score a partial state, not only a terminal one, via an LLM critic, a stepwise verifier, or a process-grade signal. When the task decomposes into a sequence of small decisions and an early bad choice is recoverable by re-exploring siblings. When you can afford to pause, score, and fork rollouts. Math proofs, multi-step code, multi-hop QA, web search, planning.

When NOT to use it

When partial states are uninformative and the only signal arrives at the terminal answer. When a single call already solves the task cheaply, the tree bookkeeping costs more than it saves. When the API is stateless completions only and you can't pause and continue rollouts. When a flat ensemble with a verifier is already enough; use Pattern 1 instead.

Concrete instances

rStar-Math: multi-seed independent MCTS trees with terminal-grounded Q-backprop

grade-school to AIME competition math· 7B SLM matches o1 on competition math· Claude / GPT / Gemini (training-free variant)
How this harness uses it

Run K independent MCTS trees with different random seeds. Within each tree, expand candidate next-steps and back-propagate Q values from terminal correctness, not from visit count or self-confidence. Scale rollouts adaptively for harder problems (8, 16, 64, 128). The first 32 candidates already diverge across trees, so by rollout 50 each tree is exploring a different subtree topology. Collect the best trajectory from each tree into one cross-tree pool, and pick the trajectory whose path has the highest mean Q.

Problem it solved here

MCTS commits to a tree shape early: hard problems whose answer lives in a low-prior region may never get explored, no matter how deep one tree goes. Multi-seed trees buy breadth: different starting hypotheses that depth alone can't reach. Terminal-grounded Q-backprop keeps the scoring honest, an intermediate node's value is set by how often following it through reached a correct final answer, not by the model's opinion of its own answer.

LATS: single tree, UCT-selected child expansion, prompted-value leaf scoring

HotPotQA (multi-hop QA) · HumanEval/MBPP (code) · WebShop (interactive web)· consistent gains across all three with one framework· GPT-4 (frozen API, no training)
How this harness uses it

One MCTS tree. At every internal node, sample k children from the same frozen policy at non-zero temperature, so siblings actually differ. Each child is scored with the prompted-value function, a second prompt to the same model asking for 1-10 on partial-trajectory correctness. UCT picks which child to expand next, combining the value score with an exploration bonus for under-visited branches. The chosen child becomes the new partial solution; recurse until a terminal trajectory is reached.

Problem it solved here

With no value function, tree search degenerates into uniform random rollouts. With one value but no sibling branching, you have a chain not a tree. The cross-sibling aggregation lets the same model's noisy individual judgements average into a usable next-step signal, while UCT keeps the search from collapsing onto whichever child got lucky on its first score.

 scores how promising each next step looks (1-10)."]

Particle Filtering: a population of partial rollouts, resampled at checkpoints

math / reasoning / code· sequential Monte Carlo applied to LLM inference· any frozen LLM
How this harness uses it

Launch N particles, each a partial rollout from the same starting prompt. At a checkpoint, every k reasoning steps, or after each tool call, pause all particles and score each partial state with a critic LLM or a process signal. Resample with replacement, weighted by the scores: low-scoring particles disappear, high-scoring particles get cloned into the empty slots. The clones diverge again because subsequent token sampling is stochastic. Repeat at the next checkpoint until particles terminate. Final answer is voted among the survivors or picked by score.

Problem it solved here

Flat best-of-N pays full compute for every rollout, including the ones that wandered off in the first step and never recovered. Particle filtering kills those rollouts and gives their budget to the survivors. The total compute is the same; the allocation happens mid-flight rather than being committed up front. Unlike MCTS, there is no tree memory, no Q-backprop, no UCT bookkeeping. Just a flat population reweighted at checkpoints. The simplest form of mid-flight reallocation.

Pattern 5

Multi-Agent Debate

Run the same question through N independent LLM instances; show each one what the others said; have each revise; majority-vote the final round. Cross-conditioning surfaces errors a single agent locks into — but only if peers are framed as opinions, not authorities, and only if you cap rounds before they collapse.

What it helps with

Questions where a single strong agent usually gets the right answer but sometimes locks into a confident error that more sampling alone can't fix. Debate exposes inconsistencies between independently-sampled reasoning traces and gives the model a chance to update on its peers' chain-of-thought. Strongest gains on factuality (biographies 66→74), MCQ reasoning (MMLU 64→71), and arithmetic / GSM-style math (GSM8K 77→85), see Du et al., 2023.

When to use it

Verifiable-answer tasks where the right answer is the kind of thing N agents can converge on (math, MCQ, factuality QA, chess move validity). Enough compute headroom for N agents × R rounds (canonical is N=3, R=2 — so ~6× the call count of a single-pass baseline). When agents are roughly comparable in capability; a wildly stronger agent gets diluted by plain majority vote.

When NOT to use it

Easy questions where a single agent already does well — debate sometimes flips correct answers wrong (iMAD's GSM8K result: vanilla MAD 76.4% vs. CoT 71.3%; the always-debate harness is below a smarter "should we debate?" gate). Open-ended generation with no answer to converge on. Tight latency budgets. Setups where the strongest agent is much stronger than the others.

How this harness uses it

Problem it solved here

Plain N-sample majority vote (self-consistency) is correlated: N independent calls to the same model on the same prompt produce N near-identical traces. MAD-M²: insert a "mask out untrustworthy peers" step between rounds

MATH · AIME24/25 · GSM8K· MATH 71.2→80.2 (+9.0), AIME24/25 6.7→13.3 (+6.6)· Qwen2.5-Math-7B-Instruct (frozen)
How this harness uses it

Same canonical MAD skeleton (N agents, R rounds, majority vote). The patch: before round r (for r≥1), each agent independently constructs a binary mask over its (N−1) peers' round-(r−1) responses, and drops the masked ones from the round-r debate prompt. Two strategies, one toggle:

  • Subjective: prompt the agent itself to label each peer YES / NO / NOT_SURE. NOs (and optionally NOT_SUREs under "strict") are masked. One extra LLM call per peer per round.
  • Objective: compute the agent's perplexity on each peer's response; mask high-perplexity (low-confidence-under-this-model) responses. No extra LLM call.
Problem it solved here

Vanilla MAD has an unbounded error-propagation pathology: a wrong response in round r is in everyone's round-(r+1) context, where it can flip them, and so on. By round R, the population's reasoning is dominated by the loudest early error. The mask cuts the propagation graph — once a peer response is flagged as suspect, it stops infecting downstream rounds. Empirically, objective (perplexity) masking is the win on hard math with a strong base model (+9 points on MATH); subjective masking helps weaker models where the LLM's self-judgment is more reliable than its perplexity calibration. Note the wart: "if the agent already knows which answer is wrong to mask, then it doesn't need debate" — masking risks discarding genuinely useful dissent on questions where the cluster's collective signal is what's correct.

iMAD: a cheap gate that decides whether to debate at all

MEDQA · MMLU · GSM8K · OKVQA · VQA-v2 · ScienceQA· GSM8K 71.3 (CoT) → 84.8 (iMAD) vs 76.4 (always-debate MAD); ~25–35% of MAD's tokens· Various base LLMs (frozen); gate-MLP trained once on PubMedQA+GQA
How this harness uses it

Three-stage pipeline. (1) Self-critique — one LLM call elicits the model's answer + a deliberate counterargument + confidences for both sides. (2) Cheap feature gate — extract 41 hand-crafted features (3 surface + 6 readability + 3 syntactic + 9 POS + 20 uncertainty lexical cues like "might", "however", "perhaps") and pass through a 6-layer MLP (no second LLM!) trained on out-of-distribution PubMedQA+GQA data with a FocusCal loss (asymmetric focal + confidence-penalty + ECE calibration term). (3) Debate only on low-confidence questions — if MLP p_correct < 0.7, run a 3-role (Affirmative / Negative / Judge) MAD with up to 5 rounds and an early-exit when the Judge emits Preference: Yes in JSON; otherwise return the self-critique's answer immediately.

Problem it solved here

Vanilla MAD has two failure modes iMAD targets at once. Wasted tokens: most questions don't need debate; an N=3, R=5 debate is 15× the call count of a single-pass baseline, and on ~66% of MEDQA questions the gate skips debate with no loss. Actively harmful debate: on GSM8K, vanilla MAD scored worse than single-pass CoT (76.4 vs 71.3) because the cross-conditioning flipped correct answers wrong on easy problems — the gate fixes this by not debating those at all (iMAD 84.8 vs MAD 76.4). The gate's net-benefit accounting (8-outcome confusion table on MEDQA) shows ~96% of decisions are net-beneficial: 5% false negatives (skipped a beneficial debate) and 1% false positives (debated and ruined a correct answer). The 41-feature MLP is essentially free at inference; the bet is that hesitation lexical cues in the model's own critique generalize across domains better than per-dataset confidence-threshold tuning.

Pattern 6

Per-Problem Routing

Run a cheap classifier on the problem before the LLM sees it; dispatch each problem to its own sub-harness tailored to that category. The router doesn't need to be smart — it needs to be cheap, stable, and well-correlated with how the right harness for this problem differs from the wrong one.

What it helps with

Heterogeneous problem distributions where the optimal recipe (retrieval, prompt, strategy, tool stack) varies meaningfully across problem categories. A single one-size-fits-all harness gets averaged into mediocrity; per-category recipes let each sub-distribution see a tuned operating point. Especially useful when category is more legible from surface features (keywords, syntax, schema) than from the LLM's own embedding.

When to use it

Tasks where (a) the problem distribution decomposes into a small number of qualitatively distinct sub-distributions (mathematical sub-fields, query intents, programming-task families) and (b) the right harness for each differs in concrete knobs you can tune separately — retrieval recipe, prompt template, tool menu, sampling budget. Especially powerful when category is observable up front rather than only knowable after solving.

When NOT to use it

Homogeneous problem distributions where a single harness already serves everything well. When the category boundary is fuzzy and the router will mis-classify often — you'll degrade more than you gain. When training a router would cost more engineering than the gain justifies; for one-off prototypes, picking the best fixed harness and shipping is the right answer.

PRISM: a small router LM predicts a soft distribution over four reasoning strategies; compute scales with the router's confidence

MATH500 · GSM8K · AQUA-RAT · SVAMP · ASDiv· +8.8 avg Pass@1 over best single baseline on Qwen2.5-Math-7B (61.0→70.7); +7.6% on Llama-3-8B· Base solver LLMs frozen; only a ~1.5B router is trained on a ~13K-instance MathStrat corpus
How this harness uses it

A small Strategy Adapter (SA, ~1.5B parameters) reads the problem text and emits a softmax distribution p ∈ R⁴ over four reasoning strategies: NLR (plain CoT), CAR (Program-of-Thoughts / PAL), TIR (tool-integrated reasoning, calculator + symbolic solver loops), and EBR (self-consistency ensemble). The harness then uses two thresholds (τ_c=0.4, τ_a=0.08) to pick one of three execution modes:

  • Confident (p_max ≥ τ_c AND margin ≥ τ_a): run only the argmax strategy.
  • Deliberative (p_max ≥ τ_c AND margin < τ_a): run top-2 strategies; majority vote.
  • Exploratory (p_max < τ_c): run all four strategies; majority vote with router-confidence tiebreak.

The router is trained with KL divergence to soft targets built from a 3-dim score per (problem, strategy): correctness, automated process quality, and per-problem-min-max-normalized efficiency (latency + length). Solver LLMs are frozen; only the router is trained, so PRISM bolts onto any pretrained solver without fine-tuning it.

Problem it solved here

Mathematical reasoning strategies are not uniformly best: PAL crushes ASDiv (86.3) but tanks on AQUA-RAT (18.1); ToRA wins AQUA-RAT (47.2) but loses MATH (41.4 < PRISM's 46.2). Fixed-strategy harnesses pay the worst-strategy tax on the off-distribution problems. PRISM's per-problem dispatch + confidence-graded compute spend lets each sub-distribution see its strongest solver, and the three-tier mode gating means easy problems consume one strategy while hard ones get the full quartet. The router learns to scale compute with difficulty: on MATH500 most problems route to Exploratory (low p_max); on ASDiv/SVAMP most route to Confident.

Pattern 7

Harness as Search Variable

Don't design the harness; search it. An outer loop treats the harness's source code as the optimization target and uses an LLM proposer to propose new variants, scored on a held-out set. The base LLM is frozen; the scaffolding is what evolves. The result is a harness no human would have written by hand.

What it helps with

Squeezing additional performance out of a frozen base LLM by optimizing the code surrounding the model call — retrieval recipes, prompt templates, refinement loops, voting strategies, tool stacks. Most of the variance in end-task quality lives in the harness, not the model weights; harness search converts engineer-hours-spent-tuning into compute-hours-spent-proposing-and-scoring. Especially powerful when the proposer is itself a strong coding agent.

When to use it

Mature benchmark with a stable evaluator, where the inner-loop harness is cheap to evaluate (so the outer loop can score many candidates) and where the search space has enough structural variation to reward exploration (different retrieval depths, different prompt template families, different aggregation rules). Frozen base model is a feature, not a bug — it makes the search target the harness, not the model.

When NOT to use it

One-shot deployments where you don't need a Pareto frontier — pick a reasonable harness and ship. Benchmarks where the search-set/test-set firewall would leak (because the proposer reads scores). Tasks without a clean numeric evaluator (open-ended generation) — the search loop needs a fitness signal. Setups where the base model is itself in flux; you'd be searching against a moving target.

Pattern 8

Decompose-and-Recurse: The Harness Calls Itself

When the input is too long or the task is too open-ended for one call, the harness lets the model spawn smaller LLM calls on chunks, sub-questions, or sub-tasks, then aggregates their results back at the root. The same model is its own subroutine, and the decomposition is discovered at inference time by the model rather than pre-specified by the engineer.

What it helps with

Inputs that exceed the context window (long documents, full codebases, dense corpora), and open-ended tasks that don't fit a fixed pipeline (research, multi-faceted questions, agentic coding). The harness scales beyond the model's native context by letting the model decide what to look at next instead of compressing everything in advance.

When to use it

When the input has structure the model can navigate at a coarse level (table of contents, file tree, list of sources), so the model can pick what to read or delegate. When the task naturally decomposes into independent or near-independent sub-problems. When you'd rather pay for many small focused calls than truncate or summarize a single huge one.

When NOT to use it

When the input is genuinely entangled and any chunk loses critical cross-references (a proof where every line depends on the prior). When the cost of orchestration (extra round trips, summarization losses, child-context spin-up) exceeds the cost of just paying for a larger native context. When a fixed pipeline already captures the decomposition cleanly, use that instead.

 or a complex open-ended task"]:::muted

 table of contents, file tree, sub-questions"]
    decide{"Decide which chunks or sub-tasks 
 need their own focused call"}:::accent
    sub1["Sub-call 1: 
 focused on chunk A"]
    sub2["Sub-call 2: 
 focused on chunk B"]
    subN["Sub-call N: 
 focused on chunk ..."]
    leaf["Each sub-call may itself recurse 
 if its chunk is still too big or too open"]:::note
    agg["Root receives summaries from each sub-call. 
 The big input never enters root context."]:::accent
    why["Key knob: the model picks the decomposition at inference time, 
 not the engineer at design time. 
 Depth and breadth of the tree are adaptive."]:::note

    root --> nav
    nav --> decide
    decide --> sub1 & sub2 & subN

    sub2 --> agg
    subN --> agg
    leaf ~~~ sub2
    why ~~~ decide

Recursive Language Models: long input as a Python variable, model calls itself on chunks

open-ended long-context QA, agent benchmarks up to 10M tokens· outperforms GPT-5 by 26% vs. compaction, 130% vs. CodeAct sub-calls, 13% vs. Claude Code· GPT-5, Claude, post-trained Qwen3-8B
How this harness uses it

The long prompt is dropped into a Python variable inside a REPL the model controls. The root model writes Python to peek at the head, tail, length, or a regex match over the variable, then calls rlm_agent(query, chunk) recursively on whichever slices it judges relevant. Each child call sees only the chunk it was given; the root never holds the raw long input in its own context. Depth and breadth of recursion are chosen by the model at each level based on what it observes.

Problem it solved here

Compaction-based long-context handling (summarize first, then answer) loses any information the summarizer didn't think mattered. Single-shot RAG retrieves what the embedder thinks is relevant, not what the model would have looked at if it could read the index itself. RLM lets the model do the retrieval recursively, with its own evolving query in control of which chunks come back into context, and the child calls' returned answers act as a compressed memo the root can keep in its working set.

Claude Code subagents: spawn child Claudes with isolated contexts and a focused task

software engineering across whole repositories· production tool with millions of users· Claude Sonnet / Opus, 1M-token main context
How this harness uses it

The main Claude session has an Agent tool that spawns a fresh sub-Claude with a single focused task and an isolated context window. The sub-agent has access to the same tools (Read, Grep, Bash, Write) on the same filesystem, runs its own gather-and-reason loop, and returns one paragraph of summary text to the parent. The parent never sees the sub-agent's intermediate greps, file reads, or tool errors. Multiple sub-agents can be launched in parallel for independent sub-tasks.

Problem it solved here

A flat agent loop pollutes its own context with every tool call's raw output, leaving less room for the actual reasoning and degrading attention over long sessions. Sub-agents act as a context boundary: the parent delegates "audit the test suite" or "find every caller of function X", and gets back a one-paragraph answer instead of a fifty-file transcript. Sub-agents can themselves spawn sub-agents, so the recursion adapts to repository scale without the engineer wiring the tree shape by hand.

Yunque DeepResearch: hierarchical research agents, each sub-agent owns one sub-question

open-ended scientific and web research, multi-source synthesis· hierarchical agentic framework, modular sub-agent orchestration· frozen LLM API
How this harness uses it

A top-level research agent receives a complex question, decomposes it into sub-questions, and dispatches each to a specialized sub-agent. Each sub-agent runs its own search loop (query, browse, extract, summarize) and may itself decompose further if its sub-question is still too broad. The top-level agent receives only the sub-agents' synthesized answers; raw browsed content stays inside the sub-agent contexts and is discarded when they return. Depth and breadth of the tree are user-controllable or model-controlled.

Problem it solved here

A flat search loop with one context window has to choose at each step between exploring more sources and reasoning over what it has, and long sessions choke on accumulated raw search results. Decomposing into sub-agents whose contexts can be discarded after they return their summary keeps the top-level reasoning context clean while letting the system explore far more sources than a single context would fit. The hierarchy of contexts plays the same role here that a tree of files plays for RLM and a parent/child agent split plays for Claude Code.

Pattern 9

Skill Retrieval: K Nearest Examples

Build up a library of validated artifacts (skills, exemplars, code snippets) across attempts. At each new task, retrieve the K most relevant entries by embedding similarity and inject them into the prompt as few-shot demonstrations or callable utilities. The prompt is dynamically assembled from prior wins, instead of being hand-curated once and frozen.

What it helps with

Distribution shift across tasks where a single fixed prompt has to cover too much, agentic systems that face open-ended task streams (lifelong agents, repo-spanning code assistance), and domains where past solved cases carry transfer signal but a static example set goes stale fast. KNN injection lets the prompt specialize to each incoming task without hand-curating per-task examples.

When to use it

When you have, or can build, a growing stream of successful prior runs that share structure with future tasks. When tasks differ enough that a one-size prompt drags performance down, but they're embeddable enough that nearest-neighbor retrieval picks up real similarity. When the retrieval substrate (vector store, BM25, hybrid) is cheap relative to the model call it shapes.

When NOT to use it

When the task distribution is narrow enough that a hand-curated few-shot set wins. When prior runs are too sparse for retrieval to find real neighbors, noise dominates. When embedding similarity correlates poorly with solution similarity (lexically similar problems with structurally different answers), KNN actively misleads. When the library admission step is unverified, wrong answers accumulate and the library poisons future calls.

    lib[("Skill library: 
 validated artifacts from prior tasks, 
 each indexed by an embedding")]:::muted
    task["New incoming task or state"]:::accent
    embed["Embed the task or state"]
    knn["Retrieve the K nearest entries by similarity"]:::accent
    inject["Inject the K entries into the prompt: 
 as few-shot examples, callable functions, 
 or reference solutions"]
    llm["LLM call, prompt now task-specific"]:::accent
    result["Result"]
    gate["Verifier admits the result to the library 
 only if it passes a check"]:::note
    grow[("Library grows")]
    why["Key invariants: 
 verifier-gated admission keeps the library clean, 
 embedding choice decides what 'similar' means."]:::note

    lib --> knn
    task --> embed
    embed --> knn
    knn --> inject
    inject --> llm
    llm --> result
    result --> gate
    gate --> grow
    grow -. "next task" .-> lib
    why ~~~ gate

Voyager: lifelong Minecraft skill library, retrieved by task-embedding similarity

Minecraft open-world tasks, lifelong learning, no terminal state· 3.3x more unique items, 15.3x faster tech-tree progress vs. prior open-source agents· GPT-4 (frozen API)
How this harness uses it

Each verified successful skill (a Mineflayer JavaScript function such as "craft iron pickaxe" or "fight zombie") is added to the skill library along with an embedding of its natural-language description. On a new task, the curriculum agent embeds the current sub-task and retrieves the K nearest skills from the library, then injects them into the action-generation prompt as callable functions the agent can compose. Admission is gated by a self-critic that has to certify the skill ran cleanly in the environment, so the library grows monotonically with verified content.

Problem it solved here

A general-purpose Minecraft agent prompt either has too few examples (the agent re-derives common skills every time it needs them, burning steps) or too many (the prompt is bloated and the agent loses the relevant ones in the noise). KNN retrieval gives the agent the K skills most relevant to the current sub-task, and only those. The library persists across episodes, so the agent inherits the work of every prior run, and the curriculum agent can target sub-tasks just beyond what the library already covers.

HASP: Skill Programs retrieved per step, intervene before the next action

long-horizon agent tasks across domains· retrieval-gated skill intervention at inference, post-training, and self-improvement· frozen LLM
How this harness uses it

HASP maintains a library of validated Program Functions (PFs), small executable skills the agent can call. At each step of the agent loop, a retrieval layer pulls the top-K relevant PFs for the current state and lets them intervene before the next LLM action is taken. The agent's prompt at each step contains a curated subset of callable utilities chosen for the situation it's in right now, not a fixed tool roster baked in at design time. The same library can be evolved via teacher-reviewed self-improvement loops.

Problem it solved here

A fixed tool roster grows unwieldy as the agent's capability surface expands, blowing out the prompt and slowing inference. KNN retrieval over PFs keeps the per-step prompt budget bounded while still giving the agent access to a large skill surface. Verified-PF gating prevents the library from drifting; the retrieval layer decides which subset to surface at each step, so a million-skill library still costs the same context budget as a hundred-skill one.

 from the PF-shaped action surface"]:::accent

SkillFlow: cascaded retrieval pipeline, embeddings narrow then rerankers sharpen

large-scale agent skill retrieval· bi-encoder, then shallow reranker, then deep reranker, then LLM selector· frozen LLM
How this harness uses it

SkillFlow's retrieval is a cascade. A bi-encoder pulls ~1,000 candidate skills from the full library, a shallow cross-encoder reranks to ~100, a deep cross-encoder reranks to ~10, and an LLM-based selector picks ≤5 to inject into the agent's prompt. Each stage trades latency for precision: cheap embeddings narrow the field, expensive rerankers sharpen the top. The final selection is the few-shot demonstrations or callable utilities that go into the next LLM call.

Problem it solved here

A single bi-encoder retrieval over thousands of skills returns too much noise to inject directly; a single cross-encoder over all candidates is too expensive to run per step. The cascade gives precision at the top of the ranked list without paying cross-encoder cost on every candidate, scaling KNN injection to libraries large enough to be useful for production agents. The structural move is the same KNN-few-shot pattern as Voyager and HASP, but the retrieval is its own multi-stage pipeline rather than a single embedding lookup.
